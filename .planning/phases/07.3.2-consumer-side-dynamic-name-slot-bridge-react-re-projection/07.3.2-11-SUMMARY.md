## Surfaced Follow-Up: F-07.3.2-11-A — Angular Consumer-Side Dynamic-Name Slot Dispatch

### What's broken

The Angular target's consumer-side emit for `<template #[$data.slotName]>` produces (verbatim from `tests/dist-parity/fixtures/ModalConsumer.angular.ts:30-32`):

```html
<rozie-modal [open]="open2()" (openChange)="open2.set($event)">
  <ng-template #__dynSlot_0>
    <span class="dynamic-fill">Dynamic header via slotName</span>
  </ng-template>
  <ng-template #defaultSlot>Dynamic-name demo body</ng-template>
  <ng-container *ngTemplateOutlet="templates[slotName()]"></ng-container>
</rozie-modal>
```

The `<ng-container *ngTemplateOutlet="templates[slotName()]">` is rendered as a CHILD of `<rozie-modal>`. For Angular to project that child into the Modal's body, the producer (`Modal.angular.ts`) would need an `<ng-content>` element. It does not — and CAN'T, because the dynamic-name dispatch is meant to populate the producer's `templates()` INPUT signal (Plan 03), which the producer's r-if guards consume via `(headerTpl ?? templates()?.['header'])`. The producer has no `<ng-content>`, so Angular silently drops the un-projected `<ng-container>` child, and the producer's `templates()` signal stays empty → `templates()?.['header']` returns undefined → both the outer r-if guard AND the inner @if guard short-circuit → `*ngTemplateOutlet` for `__dynSlot_0` never fires → `.dynamic-fill` is absent from DOM.

### What the correct emit should be

The consumer should bind `templates` as a property INPUT on the producer tag:

```html
<rozie-modal [open]="open2()" (openChange)="open2.set($event)" [templates]="templates">
  <ng-template #__dynSlot_0>
    <span class="dynamic-fill">Dynamic header via slotName</span>
  </ng-template>
  <ng-template #defaultSlot>Dynamic-name demo body</ng-template>
</rozie-modal>
```

Where `templates` is the consumer's class getter (already correctly emitted at `ModalConsumer.angular.ts:62-64`):

```ts
get templates(): Record<string, TemplateRef<unknown>> {
  return { [this.slotName()]: this.__dynSlot_0! };
}
```

### Where the fix lives

`packages/targets/angular/src/emit/emitTemplateNode.ts:407-419` — the loop that builds `dispatchParts`. Instead of appending an `<ng-container *ngTemplateOutlet="templates[...]">` as a child, the emitter needs to:

1. Compose the dynamic-templates map expression (e.g., `{ [slotName()]: __dynSlot_0 }`) inline OR via a class getter.
2. Bind it to the producer tag as `[templates]="<expr>"` (a property binding on the wrapper element).
3. Remove the `<ng-container *ngTemplateOutlet="templates[...]">` child emit entirely.

This is a structural change to the consumer-side dispatch contract — Rule 4 (architectural) work, requiring its own dist-parity re-bless of ModalConsumer.angular.ts + unit tests at the emitTemplateNode/emitSlotFiller layer + likely changes to emitSlotFiller.ts (its existing docs at L133+ describe the current `<ng-container *ngTemplateOutlet="templates['__dynSlot_<N>']">` shape, which is the broken pattern).

### Why it surfaced now (not in Plan 10)

Plan 10 closed the PRODUCER side of Angular's `$slots.X` guard merge — its dist-parity Modal.angular.ts fixture correctly emits `@if ((headerTpl ?? templates()?.['header']))`. Plan 10's own unit tests + dist-parity tests all verify this shape (which is correct).

Plan 10 deliberately deferred the Playwright cell run to the orchestrator (decisions[3] in its SUMMARY: *"Did NOT run the pinned Playwright Docker recipe for the Angular Modal 2 D-04 cell from this worktree...the orchestrator-validates-tests memo explicitly assigns final cross-cell validation to the orchestrator after merge."*). Plan 10's own assessment also flagged the likely root cause: *"If the Playwright cell still fails after merge, the most likely root cause is NOT this plan's rewrite/emit changes...but a separate runtime-binding issue in how the Angular templates() input signal receives the consumer-side `templates={...}` value. That would be a Plan 03 follow-up (templates signal input binding), not a Plan 10 issue."* — Plan 10's prediction was exactly correct.

This Plan 11 confirms: the producer-side merge IS correct (PASS at the dist-parity layer), but the consumer-side dispatch never delivers the `templates` map to the producer's input signal, so the merge's RHS (`templates()?.['header']`) always resolves to `undefined` at runtime.

### Suggested plan name + scope

**F-07.3.2-11-A: Angular consumer-side dynamic-name slot dispatch bridge** — fix `emitTemplateNode.ts:407-419` to bind `[templates]="<computed map>"` as a property input on the producer tag, removing the `<ng-container *ngTemplateOutlet="templates[...]">` projected-content emit. Re-bless ModalConsumer.angular.ts dist-parity. Add unit tests at emitTemplateNode/emitSlotFiller. Verify via the same pinned Docker recipe — the `angular: Modal 2 dynamic-fill` cell should PASS.

Estimated scope: ~3 tasks (RED unit test, GREEN emit fix, dist-parity re-bless + Docker verify). Similar shape to Plan 10 but on the consumer side.
