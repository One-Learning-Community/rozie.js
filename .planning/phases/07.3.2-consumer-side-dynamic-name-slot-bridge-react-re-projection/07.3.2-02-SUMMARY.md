## Decisions Made

1. **Kept merge expression INSIDE JSX braces (Pitfall 2 / A2).** Solid's compiler wraps JSX expression scope with effect-tracking accessors. The merge expression `(_props.<X>Slot ?? _props.slots?.['<x>'])` MUST stay inside every emit branch's JSX `{...}` braces so the reactive dependency on `_props.slots` is tracked. Hoisting outside (e.g., a local `const merged = _props.headerSlot ?? _props.slots?.['header'];` declaration in the emitted output) would break reactivity-on-change — Modal 2 would render on first paint but never update when `$data.slotName` changes. The Task 2 Pitfall 2 unit test is the standing regression gate.

2. **Did NOT touch shell.ts default-slot accessor (Open Question 2).** Per RESEARCH the Solid default-slot dynamic-name dispatch — consumer writes `<template #[$data.x]>` where `x === ''` resolves to the default slot — uses the `resolved()` accessor from `children(() => local.children)` (shell.ts ~L169) and is OUT OF SCOPE for Plan 02. Not exercised by the ModalConsumer dogfood; documented as a deferred limitation below.

3. **Used literal single-quoted string for `slotKey`.** The merge key is `'${slotName}'` (template string interpolating into a quoted JS string literal in the emit). `slotName` is statically known at emit time and is the IR slot name as authored in `<slot name="...">`; no runtime computation required. This matches the consumer emit shape from `emitDynamicSlotsProp` which produces `slots={{ [keyExpr]: ... }}` where `keyExpr` is the rewritten consumer expression — but the producer side only needs the static key it declared.

## Deferred Issues

**Open Question 2: Solid default-slot dynamic-name dispatch (OUT OF SCOPE — documented for traceability).** When a consumer writes `<template #[$data.slotName]>` and `$data.slotName === ''`, the dynamic-name resolution targets the producer's DEFAULT slot. On Solid, the default slot uses the `resolved()` accessor declared in shell.ts (`const resolved = children(() => local.children)`); the Plan 02 merge only handles NAMED slots (i.e., `slotName !== ''`). Per RESEARCH this case is not exercised by the ModalConsumer dogfood, so a fix is deferred to a future phase if it surfaces in a real consumer. The threat register tags this as T-07.3.2-09 with disposition `accept`.
