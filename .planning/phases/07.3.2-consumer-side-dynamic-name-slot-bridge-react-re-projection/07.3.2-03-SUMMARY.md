## Accomplishments

- **Producer-side intake symmetric to consumer-side getter.** Consumer emits `get templates(): Record<string, TemplateRef<unknown>>` at emitTemplateNode.ts:449 (Phase 07.2 work). Producer now declares `templates = input<Record<string, TemplateRef<unknown>> | undefined>(undefined);` as the receiving half. Same `Record<string, TemplateRef<unknown>>` type on both sides.
- **D-02 static-wins invariant enforced at the binding site.** Every `*ngTemplateOutlet` binding in slotted Angular components now reads `(<X>Tpl ?? templates()?.['<x>'])` — `@ContentChild` static-name path LEFT of `??`, dynamic `templates()` signal-call lookup RIGHT. `ngAfterContentInit` lifecycle ordering (Assumption A5) ensures the static path is populated BEFORE template expression evaluation on first render.
- **A7 signal-era override applied.** CONTEXT.md repeatedly recommended `@Input() templates?: Record<...>` decorator form; RESEARCH Assumption A7 corrected this to `input<T>()` signal form for Angular 19+ floor consistency. Unit test §templates-merge intake locks the override with a negative assertion (`expect(classBody).not.toMatch(/@Input\(\)\s+templates/)`).
- **Pitfall 3 contingency NOT triggered.** Angular 19's template grammar accepts the inline `(<X>Tpl ?? templates()?.['<X>'])` expression through `@analogjs/vite-plugin-angular`'s AOT step — verified by clean `pnpm --filter angular-analogjs-demo build`. No class-body getter fallback (`__merged<X>Tpl`) was needed; the more compact inline form ships.
- **D-05 byte-equivalence preserved.** Counter / SearchInput / TreeNode / CardHeader Angular fixtures (per-package snaps + dist-parity) show ZERO diff. ModalConsumer.angular.ts also unchanged (no own producer-side slots — the existing consumer-side `templates` getter still works untouched).
- **Cross-target D-SV-16 port validates the canonical pattern.** Three-part Svelte template (intake field, optional destructure rename, per-slot `??` merge) maps cleanly to Angular: (1) intake field becomes signal `input<T>()`, (2) destructure rename N/A (Angular reads `<X>Tpl` directly in template), (3) per-slot merge moves to BINDING SITE (`*ngTemplateOutlet` outletTag construction) since Angular has no producer-side script-body `$derived` analog.

## Latent Concern (Out of Scope, Documented for Future)

**Producer-name-collision risk for components that BOTH (a) declare producer slots AND (b) emit a consumer-side `templates` getter.** No current dogfood fixture triggers this — Modal/TodoList/Dropdown/Card/WrapperModal declare slots but have no dynamic-name slot fills; ModalConsumer has a `templates` getter but no own slots. A hypothetical future component that does both would emit BOTH `templates = input<...>(undefined);` (producer side, from Plan 03) and `get templates(): Record<string, TemplateRef<unknown>> { ... }` (consumer side, from emitTemplateNode.ts:449) on the same class → TS duplicate-name error.

Defer mitigation: Plan 03 honors the plan's narrow scope (producer-side intake only). When/if this collision surfaces (likely in a future deep-composition phase or via dogfood), the resolution is to rename one side — e.g., the consumer-side getter to `_dynamicTemplates` with a corresponding update to the consumer template emit at `emitSlotFiller.ts:180`. Tracked here so the next planner can pick it up.
