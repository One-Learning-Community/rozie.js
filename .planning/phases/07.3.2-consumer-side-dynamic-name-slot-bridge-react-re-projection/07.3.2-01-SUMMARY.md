# Phase 07.3.2 Plan 01: React Producer-side Dynamic-name Slot Intake Summary

Port the canonical Svelte D-SV-16 producer-side dynamic-name slots intake pattern to the React target. The React producer now reads the consumer-emitted `slots={{ [expr]: (ctx) => (<>...</>) }}` prop and merges it with statically-named render props at every slot invocation site via left-precedence `??`. Closes the Modal-2 (dynamic-name fill) dogfood blocker on the React consumer demo.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Pre-existing emitTemplate.test.ts assertions on bare fieldRef shape**
- **Found during:** Task 1 source compile / pre-Task-2 test triage.
- **Issue:** Two `it()` blocks in `packages/targets/react/src/__tests__/emitTemplate.test.ts` (Slot lowering Test 1 + Test 4) explicitly asserted the old bare-prop-ref emit shape: `{props.children}` and `{props.renderHeader}`. Plan 01's emitter change replaces these with the parenthesised merge `(props.children ?? props.slots?.[''])` / `(props.renderHeader ?? props.slots?.['header'])`, so the old assertions broke.
- **Fix:** Updated both assertions to match the new merged-fieldRef shape, preserved the test names with a `(Phase 07.3.2 merge)` suffix, and added inline comments citing D-SV-16 + D-18 empty-string sentinel rationale.
- **Files modified:** packages/targets/react/src/__tests__/emitTemplate.test.ts
- **Commit:** e9ab6db
- **Note:** Test 6 (named slot with params, `props.renderTrigger?.(ctx)`) used `toContain('props.renderTrigger')` substring match and continued passing without modification — the merge expression still contains the substring.

**2. [Rule 1 - Bug] Hand-tuned .compiled.tsx surface mismatch with newly-blessed snap**
- **Found during:** Per-package snap re-bless (Task 2 part B).
- **Issue:** `Modal.compiled.tsx` and `Dropdown.compiled.tsx` (hand-tuned React modules used by behavior tests under happy-dom/StrictMode that intentionally diverge from the canonical emitter output per the `compiled-fixtures-drift.test.ts` header) lacked the new `slots?:` field. The drift test asserts propsFields set-equality between .compiled.tsx and .tsx.snap; the new field on the snap broke set-equality.
- **Fix:** Added a surface-only `slots?: Record<string, (ctx: any) => ReactNode>;` field to both compiled.tsx files with a comment noting that behavior tests never exercise the dynamic slots map. The field exists purely for propsFields set-equality.
- **Files modified:** packages/targets/react/tests/integration/Modal.compiled.tsx, packages/targets/react/tests/integration/Dropdown.compiled.tsx
- **Commit:** e9ab6db
- **Rationale:** The compiled-fixtures-drift.test.ts file header documents that auto-regenerating .compiled.tsx from .snap would silently undo hand-corrections (case 3 in the header — Modal's `if (!open) return` guards under StrictMode). Adding a single surface-only field is the minimal change that satisfies set-equality without disturbing the hand-tuned behavior corrections.

**3. [Rule 1 - Bug] Plan-enumeration under-counted slotted React producers**
- **Found during:** dist-parity bootstrap (Task 2 part B step 2-3).
- **Issue:** The plan listed 5 dist-parity React files (Modal.tsx, Modal.d.ts, ModalConsumer.tsx, WrapperModal.tsx, WrapperModal.d.ts) as expected re-blesses. Running `pnpm --filter dist-parity bootstrap` re-blessed 10 React files because Dropdown.{tsx,d.ts}, TodoList.{tsx,d.ts}, and Card.{tsx,d.ts} are also slotted producers that correctly gain the new intake field per the Plan 01 emitter change.
- **Fix:** Staged all 10 affected React dist-parity fixtures plus the per-package snap re-blesses. ModalConsumer.tsx and ModalConsumer.d.ts (consumer-side) showed ZERO diff, confirming the intake field is producer-side-only as designed. CardHeader.tsx (non-slotted) showed ZERO diff. Vue/Svelte/Lit/Solid/Angular fixtures for Modal/WrapperModal/Dropdown/TodoList/Card all showed ZERO diff (D-05 cross-target invariant honored).
- **Files affected:** Added 6 fixtures to the commit beyond the plan's enumeration (Dropdown.{tsx,d.ts}, TodoList.{tsx,d.ts}, Card.{tsx,d.ts}).
- **Commit:** e9ab6db
- **Plan-vs-reality reconciliation:** The plan's D-05 wording said "Counter / SearchInput / Dropdown / TodoList etc. are byte-identical (D-05)" — internally inconsistent because Dropdown has a `trigger` slot and TodoList has `header`/`default`/`empty` slots. The correct D-05 principle (Pattern S2 in PATTERNS.md) is "non-slotted components must be byte-identical" — that principle is fully honored (Counter, SearchInput byte-equivalent). The fix decision was to follow the principle, not the enumeration.

## Threat Mitigations Honored

- **T-07.3.2-01 (Tampering — merge precedence inversion):** mitigated — unit test §slots-merge invocation explicitly asserts `(props.renderHeader ?? props.slots?.['header'])` with static LEFT, dynamic RIGHT of `??`. Provenance comment in emitSlotInvocation.ts cites D-02 + Svelte canonical commit 6060408.
- **T-07.3.2-02 (Information disclosure — inline TSX vs public .d.ts drift):** mitigated — Task 1 atomically updated BOTH emitPropsInterface.ts AND emitTypes.ts; §slots-merge intake unit test asserts BOTH outputs contain the field; Modal.d.ts dist-parity fixture re-blessed alongside Modal.tsx.
- **T-07.3.2-03 (Tampering — runtime dispatch via `props.slots?.[name]` for consumer-controlled `name`):** accepted per plan threat model — emitSlotFiller.ts emits a plain object literal; `__proto__` as a key compiles to a literal own-property, not a prototype attack.
- **T-07.3.2-04 (Tampering — D-05 violation):** mitigated — both `ir.slots.length > 0` gates verified via §slots-merge intake negative assertions (Counter has no slots field) AND via `git diff` byte-equivalence checks on Counter.tsx / SearchInput.tsx dist-parity fixtures.
