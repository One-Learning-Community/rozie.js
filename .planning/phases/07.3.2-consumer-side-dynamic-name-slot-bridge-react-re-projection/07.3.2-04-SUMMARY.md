# Phase 07.3.2 Plan 04: React WrapperModal Re-Projection — SC#4 Root Cause Closed Summary

**One-liner:** Fixed React WrapperModal Modal 3 silent-fail re-projection root cause by aligning refineSlotTypes.ts:108 with emitTypes.ts:152 (`'() => ReactNode'`) and switching emitSlotInvocation no-params named-slot path to INVOKE via `?.()` — composing cleanly with Plan 01's merge.

## What Landed

This plan closed SC#4 — the React WrapperModal re-projection bug class. Two minimal source-line changes fixed a long-standing divergence between the inline TSX `Props` interface (which declared `ReactNode`) and the public `.d.ts` (which has always declared `() => ReactNode`) for no-params named slots. Consumer-side `emitSlotFiller.ts:126` ALWAYS wraps slot bodies in an arrow function (`renderHeader={() => (<>...</>)}`); before this fix, the producer rendered the function reference directly as a React child — triggering React's "Functions are not valid as a React child" error in dev OR silent-no-op in production. WrapperModal Modal 3 happens to be the dogfood demo that exercises a 2-level wrapper chain where the divergence became user-visible.

The fix has two paired changes:

1. **`refineSlotTypes.ts:105-128`** — `propFieldType: 'ReactNode'` → `propFieldType: '() => ReactNode'` for the named-slot no-params branch, aligning the inline TSX interface with `emitTypes.ts:152`'s public `.d.ts` shape. This is the **Pattern 2 alignment** precedent — the same divergence-then-alignment pattern as the default-slot-with-params union form at L88-100.

2. **`emitSlotInvocation.ts:279-303`** — no-params named-slot path now INVOKES the function. Bare fallback-less form is `{${fieldRef}?.()}`; conditional forms are `{${fieldRef} ? ${fieldRef}() : ${fallback}}`. Mirrors the with-params style at L301-309 (`?.(paramObj)` becomes `?.()` since no params).

The two changes layer cleanly on Plan 01's merged `fieldRef`: the final emit shape is `(props.renderBrand ?? props.slots?.['brand'])?.()` — valid JS (`(a ?? b)?.()`).
