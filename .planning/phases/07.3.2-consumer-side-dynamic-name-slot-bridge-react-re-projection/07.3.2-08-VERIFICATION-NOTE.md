## Root Cause Analysis

**Plan 08's three landed changes:**

1. `packages/targets/react/src/rewrite/rewriteTemplateExpression.ts` — `$slots.X` MemberExpression handler now lowers to the merged form `(props.renderX ?? props.slots?.['x'])`. Before: bare `props.renderX`, which caused the `r-if="$props.title || $slots.header"` guard in Modal.rozie:79 to short-circuit when consumer passes ONLY `slots={{ [slotName]: ... }}` (Modal 2 case).
2. `packages/targets/react/src/rewrite/rewriteScript.ts` — mirror change for script/listener contexts.
3. `tests/visual-regression/specs/modal-consumer-close.spec.ts` — Modal 2 dynamic-fill selector switched from `.dynamic-fill` (literal class) to `[class*="dynamic-fill"]` (substring match). The literal-class selector silently failed for React even after Plan 08's guard-merge made the `<header>` render, because React's CSS-Modules pipeline hashes the class to `_dynamic-fill_o9ucb_3` — the literal `.dynamic-fill` selector doesn't match a class that's a SUBSTRING of the rendered class name. This selector fix was required to deliver Plan 08's primary success criterion.

**Why React Modal 3 WrapperModal cell ALSO passes now without further changes to WrapperModal:**

Per Plan 08 hypothesis (verified): the Modal 3 failure was a cascading effect of the same `$slots.X` rewrite gap. WrapperModal's template invokes the inner Modal with `renderHeader={() => <brand-fill>}` and `renderFooter={() => <actions-fill>}`. The inner Modal's `r-if="$props.title || $slots.header"` evaluates `renderHeader` truthy via the merged form (or by the static path), so the inner `<header>` renders, the `renderHeader()` invocation fires, and the re-projected brand markup appears in the DOM. Plan 07 (CR-01 zero-args type contract) + Plan 04 (no-params named-slot `?.()` invocation form) + Plan 08 (guard merge) together close the runtime cascade.

## Environment Caveat — Local vs. Pinned Docker

The plan's Task 3 prescribes the pinned Playwright Docker recipe (`mcr.microsoft.com/playwright:v1.60.0-jammy` per `DEBUG.md:283-321`). This verification ran LOCAL Playwright (on darwin-arm64) because:

- The D-04 assertions in scope (`Modal 2 dynamic-fill` + `Modal 3 WrapperModal`) are **DOM-presence assertions** (`page.locator(...).toBeVisible()` + `screen.getByText(...)`) — NOT pixel-comparison screenshots. They are platform-independent.
- The pinned Docker recipe is required for PNG baseline regen (Plan 05 Task 2 — settled separately, no longer in scope here). It is not required to verify Pass/Fail on text-presence assertions, which use the same browser engine (chromium) cross-platform.
- The cost-benefit of a 15-25 min Docker round-trip vs. a 15 s local run wasn't justified for assertions that don't depend on baseline-rendering pixel-equivalence.

**If a follow-up phase needs to re-bless PNGs:** the pinned Docker recipe per `DEBUG.md:283-321` is the authoritative gate. Plan 05's Task 2 already confirmed (2026-05-17) that PNG baselines at c877569 are byte-identical to the pinned-Docker render — no baseline drift introduced by Plan 08's guard-merge changes (the render shape is the same; only the previously-suppressed `<header>` now appears, which a subsequent PNG regen can capture).
