# Phase 06.4 Plan 01: Scaffold Lit Target + Runtime + Wiring Summary

P1 of the Phase 06.4 Lit/Web-Components target trilogy. Adds the @rozie/target-lit + @rozie/runtime-lit packages, wires the entire toolchain (compile → unplugin → CLI → babel-plugin) to accept `target: 'lit'`, registers ROZ830/831/840 diagnostics inside D-LIT-19 Option A's reallocated tail, and ships 5 functional runtime helpers under happy-dom unit tests. The emitLit orchestrator is a one-line stub returning `export default class STUB {}\n` for any input — P2 replaces the stub body with real Lit class emission. P3 lands the consumer demo + dist-parity.

## Objective Restatement

Stand up the full Lit target scaffolding so `compile(source, { target: 'lit' })` produces *valid* (parseable, but not yet behaviorally complete) `.ts` output for all 8 reference examples, every public surface is reachable (compile API → unplugin → CLI → babel-plugin), every diagnostic code is registered, every package builds, and the runtime helpers are functional.

## Key Decisions Made

1. **D-LIT-19 Option A — ROZ code allocation**: ROZ830 (UNPLUGIN_LIT_PEER_DEP_MISSING), ROZ831 (UNPLUGIN_LIT_SIGNALS_PEER_DEP_MISSING), ROZ840 (RUNTIME_LIT_CONTROLLABLE_MODE_FLIP) registered inside the formerly-reserved tail of the babel-plugin range (ROZ820..ROZ849). Babel-plugin keeps ROZ820..ROZ829; Lit owns ROZ830..ROZ849. CLI ROZ850 untouched. The plan's must_have explicitly warned that ROZ850 was already taken (CLI_INVALID_TARGET) — picking 840 over the original 850 sketch was load-bearing.

2. **VIRTUAL_SUFFIX_LIT collision strategy**: The Lit virtual suffix `.rozie.ts` is byte-identical to `VIRTUAL_SUFFIX_ANGULAR`. Decided to declare two named constants (one per target) sharing the same string, with runtime dispatch keyed on the closure-captured `target` value at hook construction time. Only one Rozie config instance can be active per Vite invocation, so the collision is benign. Documented in transform.ts comments.

3. **attachOutsideClickListener + injectGlobalStyles in @rozie/runtime-lit (not inlined per-component)**: Per Claude's Discretion. P2 will validate emitter output imports the helpers from `@rozie/runtime-lit` rather than emitting inline IIFEs per example. This keeps emitted .ts files small and DRY.

4. **emitLit stub return value**: `{ code: 'export default class STUB {}\n', map: null, diagnostics: [] }` for ANY input. Sufficient for P1's "non-empty .ts string for all 8 examples" success bar. P2 replaces with real orchestration.

5. **No `detectLitPlugin`**: Lit has no host Vite plugin (unlike vite-plugin-solid / @sveltejs/vite-plugin-svelte / @analogjs/vite-plugin-angular). Lit components are plain ES modules that self-register via `customElements.define()` at module load. `assertLitPeerDeps` therefore only checks the two consumer deps (`lit`, `@lit-labs/preact-signals`).
