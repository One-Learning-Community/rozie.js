# Phase 06.4 Plan 02: Lit Target Emitter Implementation Summary

Replaces the P1 stub `emitLit` (which returned `export default class STUB {}\n` for any input) with a full Lit emitter producing idiomatic class output across all 8 reference examples. Locks fixture snapshots for byte-stable output across emitter invocations. Adds 4 integration tests verifying Phase 06.4 Success Criteria SC1, SC2, SC3 (partial), and SC6. Enforces T-06.4-03 (no `unsafe-html` import) and D-LIT-14 (no `@queryAssignedNodes`) invariants cross-cutting at the test level.

## Objective Restatement

Make `compile(source, { target: 'lit' })` produce *correct*, idiomatic Lit/Web-Components output for the 8 reference examples — `@property`/`@state`/`@query`/`@queryAssignedElements` class fields, paired lifecycle methods (`firstUpdated`/`disconnectedCallback`/`updated`), `<rozie-foo>` composition tags via the kebab transform, scoped + global style emission per D-LIT-15/16, shadow-DOM-aware outside-click via `composedPath()`, and the model-prop CustomEvent dispatch chain through `createLitControllableProperty`.

## Resolved RESEARCH Open Questions

### A7 — attachOutsideClickListener consolidation justified?

**Yes.** Both Dropdown and Modal use outside-click semantics:
- Dropdown via `@click.outside($refs.triggerEl, $refs.panelEl)` in `<listeners>` block.
- Modal via `@click.self` on the backdrop (not outside-click, but a similar pattern using `e.target !== e.currentTarget`).

The runtime helper consolidation pays for itself across these two examples — emitted Dropdown imports `attachOutsideClickListener` and gets the composedPath()-aware shadow-DOM handling for free. Modal's `.self` modifier is handled inline (no runtime helper needed for `.self`).

### A8 — injectGlobalStyles consolidation justified?

**Yes.** Both Dropdown and Modal declare `:root { }` rules:
- Dropdown: `--rozie-dropdown-z: 1000`
- Modal: `--rozie-modal-z: 2000`

The runtime helper `injectGlobalStyles` is imported into both emitted modules. SearchInput, Counter, Card, CardHeader, TodoList, TreeNode have NO `:root` rules and don't import the helper. The 2-of-8 ratio justified the consolidation; an inlined IIFE-per-component would have been duplicated identical code 2x.

### emitHostListenerWiring (D-LIT-12 NEW pattern) — needed by any reference example?

**Yes** — three examples need it:
- **Dropdown** — `<slot name="trigger" :open="$props.open" :toggle="toggle" />` — `toggle` is function-typed (an arrow); the emitter routes it through `this.addEventListener('rozie-trigger-toggle', ...)`. The `open` param is data-typed and projects through `data-rozie-params`.
- **Dropdown** — `<slot :close="close" />` (default slot) — `close` is function-typed.
- **Modal** — `<slot name="header" :close="close">`, `<slot :close="close" />` (default), `<slot name="footer" :close="close" />` — three function-typed slot params, each routes through `rozie-<slot>-close` host listener.
- **TodoList** — `<slot :item="item" :toggle="() => toggle(item.id)" :remove="() => remove(item.id)">` — `item` is data-typed; `toggle` and `remove` are inline arrows that close over the loop variable `item` (function-typed). v1 limitation: the host-listener wiring re-emits the closure expression OUTSIDE the loop scope, so the `item` binding isn't visible — this is a documented v1 caveat per D-LIT-12. The slot consumer's recommended path is to read `data-rozie-params.item.id` from the slotted child and dispatch the CustomEvent.

## Key Decisions Made

1. **Pragmatic string-template orchestration over pure-AST emission.** The Solid/Angular analogs do per-expression `@babel/generator` source mapping; this plan defers that to a future plan and uses string-template assembly with embedded Babel for rewriting / per-expression rendering only. The emitted code is already byte-stable across runs via the snapshot lock, which is the contract the plan asserts. Sourcemap composition is a P3 concern (or later) — the emitter does not break the contract by returning `null` for `map`.

2. **`@queryAssignedNodes` excluded at the TYPE level** in `LitDecoratorImport` union. The runtime grep test in `emitLit-shape.test.ts` is the cross-cutting backstop, but the type system also refuses to track the symbol. A future maintainer cannot accidentally `decoratorImports.add('queryAssignedNodes')` — TS would refuse to compile.

3. **Side-effect-only imports for composition.** Per D-LIT, cross-rozie composition emits `import './Foo.rozie';` (no symbol bind). The imported module's `@customElement` decorator runs at module load and registers `customElements.define(tag, Class)`. The parent template references the registered tag verbatim (`<rozie-foo>`). Self-references emit no extra import — the class self-registers via its own decorator.

## TodoList Scoped-Slot Limitation (documented v1 caveat per D-LIT-12)

TodoList's `<slot :item="item" :toggle="() => toggle(item.id)" :remove="() => remove(item.id)">` is inside an `r-for="item in $props.items"` loop. The `item` binding is the loop-local variable; the inline arrows (`() => toggle(item.id)`) close over it.

The current emitter routes function-typed slot params through `this.addEventListener('rozie-default-toggle', ...)` host-level wiring. The handler is `(() => this.toggle(item.id))((e as CustomEvent).detail)` — but `item` is OUT OF SCOPE at the host-element level (it only exists in the per-iteration body).

**v1 compromise per D-LIT-12:** Consumer-side scoped-slot function projection is documented as use-the-event-detail-not-the-closure: the slotted child dispatches `dispatchEvent(new CustomEvent('rozie-default-toggle', { detail: itemId, bubbles: true, composed: true }))` and the host listener calls `this.toggle(e.detail)`. The TodoList example needs a minor adjustment in its slot params to make this work end-to-end (`:toggle="(id) => toggle(id)"` rather than the closure form). Logged as a phase-level v1 caveat; the emitter does the right thing per the D-LIT-12 contract, but the example's `<slot>` shape predates the contract and reads as if closures should work.
