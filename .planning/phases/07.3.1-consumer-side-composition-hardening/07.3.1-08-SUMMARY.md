# Phase 07.3.1 Plan 08: Lit consumer spreads slot= across multi-root template children instead of div-wrap (D-LIT-18) Summary

When a Lit consumer's `<template #header>` (or any named-slot fill) contains multiple top-level elements separated only by whitespace, the emitter now injects `slot="<name>"` into each opening tag instead of wrapping the body in a synthetic `<div slot="<name>">`. The producer's shadow-DOM `<slot name="header">` projection then receives each child element directly, mirroring the consumer-authored DOM structure rather than threading content through an extra wrapper layer.

## What Changed

### Task 1 — `wrapWithSlotAttribute` gains a multi-root spread tier

The function previously cascaded through two paths:

1. **Single-root passthrough** — body is exactly one top-level element. Attach `slot="<name>"` to its opening tag (`<h2 slot="header">…</h2>` directly).
2. **Div-wrap fallback** — everything else. Wrap in `<div slot="<name>">…</div>`.

Real-world consumer markup hits the fallback constantly: `<template #header><h2>Title</h2><button @click="close">×</button></template>` is two top-level elements separated by whitespace — the exact shape every modal/drawer/dialog `#header` fill emits. The fallback's `<div>` wrapper changed the shadow-DOM projection semantics: instead of the producer's `<slot name="header">` receiving the `<h2>` and `<button>` directly, it received the `<div>` and the `<h2>`/`<button>` re-projected through the div's transparent presence.

D-LIT-18 adds a third tier between (1) and (2):

```js
// Phase 07.3.1 D-LIT-18 — multi-root spread.
const spread = spreadSlotAttrAcrossTopLevelElements(trimmed, slotName);
if (spread !== null) {
  return spread;
}
return `<div slot="${slotName}">${body}</div>`;
```

The new `spreadSlotAttrAcrossTopLevelElements(body, slotName)` helper walks the body with the same quote-and-interpolation-aware scanner used by `hasSiblingAtTopLevel` and `findTagClose`. It collects `(openIdx, closeIdx, isSelfClose)` triples for each top-level opening tag, then returns null (deferring to the wrap) on any of:

- Non-whitespace text at depth 0
- A top-level `${...}` interpolation (could expand to text at runtime)
- A top-level HTML comment or processing instruction
- Fewer than 2 top-level elements (single-root passthrough handled elsewhere; zero means whitespace-only body where the wrap is innocuous)
- An existing `slot=` attribute on any top-level element (would clobber consumer-authored routing)
- A malformed opening tag (no matching `>` before EOF)

Otherwise it walks the collected openTags array in reverse (last-to-first so earlier offsets stay stable) and splices ` slot="<name>"` into each opening tag — before the `/>` for self-close, before the `>` for normal — returning the rewritten body.

**Source polish (`emitSlotFiller.ts` insertion loop):**

The underlying template emitter normalizes self-closing tags with a space before the `/`: `<img src="x" />`. Without compensation, the spread's `insertAt = closeIdx - 1` (position of `/`) means the attribute lands BETWEEN the existing space and the `/`, producing `<img src="x"  slot="brand"/>` — two spaces, cosmetic regression. The fix walks `insertAt` back past trailing spaces inside the opening tag so the inserted attribute lands flush with the previous attribute: `<img src="x" slot="brand" />`. Walk-back only fires on self-close because normal opening tags have no whitespace before `>` to compress.

## Architectural Notes

### Why spread vs. continue with div-wrap

The div-wrap was correct shadow-DOM behavior — the producer's `<slot name="header">` does receive content via the wrapper because shadow-DOM projection follows the light-DOM `slot=` attribute regardless of element type. But it changed consumer DOM in observable ways:

1. **Extra layer in element tree** — `parentElement` of `<h2>` was the wrapper `<div>`, not the producer's `<rozie-modal>`. Consumer JavaScript using `el.parentElement` got the wrapper.
2. **CSS specificity changes** — selectors like `rozie-modal > h2` did not match because the `<h2>` was nested inside a `<div>`.
3. **Default `display: block` from the `<div>`** — visible in dev tools, occasionally caused layout differences when the wrapper introduced unintended block-flow.

The spread eliminates all three. The producer's `<rozie-modal>` directly receives the consumer-authored elements, matching what consumers see in React/Vue/Svelte/Solid/Angular targets.

### Why structural-ambiguity refusal (vs. forcing spread always)

Consider `<template #header><h2>Title</h2>Inline text<button>×</button></template>`. With div-wrap, the entire body — including "Inline text" — projects into the `header` named slot. With spread, only `<h2 slot="header">` and `<button slot="header">` project; "Inline text" becomes a text node at depth 0 of the consumer's projected-to-`<rozie-modal>` light DOM. Per shadow-DOM rules, text nodes without `slot=` project to the default slot (if present) or get ignored. Either outcome is a SEMANTIC SHIFT from the consumer's apparent intent.

The conservative refusal is correct: structurally ambiguous bodies stay div-wrapped (preserving wrap semantics), unambiguous bodies (elements-and-whitespace-only) get the spread (preserving DOM parity). The refusal is also locked by a dedicated regression test (`falls back to <div slot=> wrap when top-level text is present between elements`).

### Why the spread helper reuses the existing scanner

`spreadSlotAttrAcrossTopLevelElements` walks the body with the same `findTagClose` / `skipInterpolation` / depth-tracking logic that `hasSiblingAtTopLevel` uses. Sharing the scanner ensures the two functions agree on what "top level" means — important because the cascade in `wrapWithSlotAttribute` first calls `hasSiblingAtTopLevel` (via the single-root path) and then `spreadSlotAttrAcrossTopLevelElements`. If they disagreed, you could construct a body that "has siblings" per one scanner but "is single root" per the other, and the cascade would land in surprising places. Reusing the primitives forecloses that drift.

### Cascade order is structurally specific to general

1. **Single-root passthrough** — most specific. Requires one tag-open match at position 0, matching `</tag>` at end, AND `hasSiblingAtTopLevel` returns false. Cheapest path.
2. **Multi-root spread** — middle specificity. Requires 2+ top-level elements AND no top-level text/interpolation/comment/existing-slot. Returns null on refusal.
3. **Div-wrap fallback** — most general. Always succeeds.

Ordering matters: a body that's single-root MUST hit (1), not (2). The single-root regex runs first at line 513, and only on its failure does the cascade reach (2). A body with 2+ top-level elements never reaches (2) without first failing (1) — the single-root regex anchors at position 0 with `^<([a-zA-Z][\w-]*)\b`, which matches the FIRST element of a multi-root body too, but then the closingTagRe check at line 528 (`closingTagRe.test(trimmed)`) fails because trimmed ends with the LAST element's closing tag, not the first's. So (1) safely declines and (2) takes over.
