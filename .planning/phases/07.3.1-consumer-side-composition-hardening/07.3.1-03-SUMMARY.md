# Phase 07.3.1 Plan 03: Lit observer first-paint fix (Blocker #3, Option D) Summary

Implements RESEARCH Option D (combined fix) for the Lit consumer-side
scoped-slot-ctx first-paint race that was deferred from Phase 07.3 closeout
as Blocker #3 (D-03). Late-binding event-handler wrap closes Race A
(render-time capture of undefined ctx); per-filler `_slotCtxWired_<name>`
flag + `queueMicrotask` retry + `updated()` re-attempt + disconnect reset
closes Race B (producer-upgrade ordering). Acceptance gate lives in Plan
04's dogfood close-spec un-fixme, but the emit-shape change ships here so
parallel Wave 1 worktrees stay disjoint.

## Tasks Completed

### Task 1: Late-binding event-handler wrap (Race A) — `feat(07.3.1):` 2d4133b

In `packages/targets/lit/src/emit/emitTemplate.ts:buildEventParts`, immediately
after `rewriteTemplateExpression(listener.handler, ir)` produces the handler
expression string, detect the consumer-side scoped-slot-ctx shape via the
regex `/this\._[A-Za-z0-9_]+Ctx\?\./` (tight per Landmine 3 — matches only
what `emitSlotFiller.rewriteScopedParamRefs` produces, not user-authored
`_xxxCtx` fields). When matched, wrap as `(e) => (${handlerRaw})?.(e)` so
the ctx read happens at click time, not render time. Without this, Lit's
`@click=${undefined}` evaluates to "no listener installed" on first render,
and even after the observer fires and `requestUpdate()` re-renders, a click
between those moments is silently dropped.

### Task 2: `_slotCtxWired_<name>` flag + microtask retry + updated() re-attempt + disconnect reset (Race B) — `feat(07.3.1):` 52426eb

Four-file coordinated change:

**`emitSlotFiller.ts`** (per scoped filler):
- Adds `private _slotCtxWired_<name> = false;` class field next to the
  existing `_<name>Ctx` field (so emitLit splices both into the class body).
- Replaces the previous silent-skip IIFE in `firstUpdatedLines` with a
  `tryWire = () => { ... }` arrow + immediate `tryWire()` + `queueMicrotask`
  fallback that re-runs `tryWire()` when the flag is still false. Inside
  the `if (slotEl)` branch, sets `this._slotCtxWired_<name> = true;`.
- Returns two new result-interface keys:
  - `updatedBodyLines`: a re-attempt block (`if (!this._slotCtxWired_<name>) { ... }`)
    inlined for the generated `updated()` method body.
  - `disconnectResetLines`: a single `this._slotCtxWired_<name> = false;` line
    appended inside the generated `disconnectedCallback()` after the
    `_disconnectCleanups` drain (Landmine 2 — without this, a re-mounted
    consumer carries the stale `true` flag and skips both retries forever).

**`emitTemplate.ts`**:
- Extends `EmitTemplateOpts._state` and `EmitTemplateResult` with
  `slotFillerUpdatedBody: string[]` and `slotFillerDisconnectReset: string[]`.
- At the existing emitSlotFiller routing site (lines 737-750 pattern), pushes
  the new arrays from each filler's emission into the corresponding
  collectors.
- Initializes both arrays in the per-call state object and surfaces both
  through the final return so emitLit can consume them.

**`emitLit.ts`**:
- Composes `templateResult.slotFillerUpdatedBody` with
  `scriptResult.updateHookBody` (slot-filler retry runs FIRST so a successful
  retry's `requestUpdate()` is observed by user `$onUpdate` hooks on the
  very next update cycle), passes the result as `parts.updatedBody`.
- Adds `slotFillerDisconnectReset: string` to `ComposeClassBodyParts`.
- In `composeClassBody`'s `disconnectedCallback()` emission, appends
  `parts.slotFillerDisconnectReset` AFTER the `_disconnectCleanups = []`
  drain — so the flag reset survives the cleanup cycle.

**Smoke test** (`smoke-d03.test.ts` — 6 cases):
1. Late-binding wrap shape: `@click=${(e) => (this._headerCtx?.close)?.(e)}`
2. `private _slotCtxWired_header = false;` class field present
3. `tryWire` + `queueMicrotask` retry inside `_armListeners()` body
4. `updated(changedProperties: ...)` body contains `if (!this._slotCtxWired_header)` re-attempt
5. `disconnectedCallback()` resets flag AFTER `_disconnectCleanups = []` drain (positional check)
6. Plain (non-scoped) handler `@click="handleClick"` stays as
   `@click=${this.handleClick}` — NOT wrapped (Landmine 3 false-positive guard)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `wrapWithSlotAttribute` regex falsely terminated at `>` inside `${...}` interpolations**
- **Found during:** Task 2 smoke test (6th test, plain-handler false-positive case)
- **Issue:** Existing `wrapWithSlotAttribute` helper used a `[^>]*` regex to
  extract the opening tag's attribute string. When the body became
  `<button @click=${(e) => (this._headerCtx?.close)?.(e)}>×</button>`, the
  `>` inside `=>` was misidentified as the tag close. The slot attribute
  was then injected at the wrong position, producing garbage emit:
  `<button @click=${(e)  slot="header"> (this._headerCtx?.close)?.(e)}>×</button>`.
- **Why this is a Rule 1 fix (not a deviation):** The Part 1 wrap shape is
  correct; the regex was a latent fragility (it also couldn't handle
  `<button title="a > b">` or any other `>` inside attribute interpolations).
  The wrap surfaced the latent bug — fixing it is required for Task 2
  acceptance.
- **Fix:** Rewrote `wrapWithSlotAttribute` to use the existing `findTagClose`
  scanner (already quote-aware) extended with a new `skipInterpolation`
  helper that walks a balanced `${...}` block (tracking brace depth and
  string-quote state). Also extended `hasSiblingAtTopLevel` with the same
  `${...}` skip. The single-root passthrough now correctly identifies the
  tag boundary regardless of `>` characters inside attribute-value
  interpolations.
- **Files modified:** `packages/targets/lit/src/emit/emitSlotFiller.ts`
  (helper functions rewritten alongside the main Task 2 changes)
- **Commit:** Bundled into Task 2 commit `52426eb` (same file, same change
  unit — splitting into a separate commit would have left Task 2 broken).

## Per-package snap re-bless: no-op

The plan's `files_modified` listed three Lit snap fixtures
(`Modal.lit.ts.snap`, `Dropdown.lit.ts.snap`, `TodoList.lit.ts.snap`) as
needing re-bless. Running `vitest run --update` after both tasks produced
**zero snap changes** — confirmed via `git status --short` showing only the
intentionally modified source files. These fixtures are compiled from
`examples/{Modal,Dropdown,TodoList}.rozie` which are PRODUCER definitions
(they declare `<slot name="header" :close="close">` etc.) and contain NO
consumer-side `<template #header="{ close }">` fillers — so the D-03 emit
changes are dormant for them. The plan's anticipatory list was accurate as
a "would-need-re-bless-IF-affected" check; in practice nothing drifted at
the per-package layer. Dist-parity re-bless (which exercises the actual
consumer-side `ModalConsumer.lit.ts` byte-locked fixture) is Plan 04's
acceptance gate.
