# Phase 07.3.1: Code Review Report

**Reviewed:** 2026-05-16
**Depth:** standard
**Files Reviewed:** 14
**Status:** issues_found

## Summary

Phase 07.3.1 closes 3 Phase 07.3 deferred blockers plus 4 newly-surfaced
architectural gaps. The dispatchEvent translation (D-LIT-17) is well-defended:
the regex character class `[A-Za-z0-9_]` on the captured slot/param names
forecloses event-name injection via the rewritten handler text. The slot=
spread (D-LIT-18) uses a quote/interpolation-aware scanner with conservative
fallback that mostly does the right thing.

The serious correctness gaps are all on the **emitter-output** side rather
than the validator side:

1. The Svelte `$derived` slot merge (D-SV-16) silently emits **duplicate
   top-level `const` declarations** when a consumer authors a slot whose name
   collides with a `<data>` / `<computed>` / script-local binding â€” most
   notably any component with both a default slot and any state/script-local
   named `children`. The emitted Svelte is a SyntaxError.
2. The Lit pre-seed (D-LIT-15) walks `this.children` for the default-slot
   text-node tolerance branch, but `this.children` is an HTMLCollection that
   excludes text nodes by definition (only Element children, all with
   `nodeType === 1`). The text-node tolerance the comment claims is dead
   code, and bare-text default-slot fills leave `_hasSlotDefault` permanently
   false â€” breaking any conditionally-wrapped default slot.
3. Slot names and prop names are interpolated into emitted CSS selectors,
   HTML attribute values, JS template literals, and field identifiers
   throughout the Lit emitter (`emitSlotFiller.ts`, `emitSlotDecl.ts`) and
   the Svelte emitter (`emitScript.ts`) **without any escaping or
   character-class validation at any layer**. A slot named `foo"]bar`,
   `2items`, `foo-bar`, or one containing whitespace silently produces
   broken output â€” querySelector throws SyntaxError, classes have
   malformed field names, etc. Documented schema or input validation
   contract is missing.

Several mid-tier concerns: defensive resolver fallbacks emit syntactically
broken assignments if the validator ever regresses (silent), `lvalue.ts`
accepts `OptionalMemberExpression` per its public contract but the
React/Lit resolver helpers only check `MemberExpression`, and the
`producerLookupHardFailed` flag in the two-way validator is set but never
read (carrying `void` as a fig-leaf).

## Critical Issues

### CR-01: Svelte `$derived` slot-merge collides with user-authored top-level bindings

**File:** `packages/targets/svelte/src/emit/emitScript.ts:266-274` (`emitSlotDerivedMerges`)
and `packages/targets/svelte/src/emit/emitScript.ts:228-232`
(`buildPropsDestructureEntries`)

**Issue:** The D-SV-16 merge emits `const ${key} = $derived(__${key}Prop ?? snippets?.${key});`
where `key` is `'children'` for the default slot or the literal slot name
otherwise. The emitted const lands at top level in `<script>`. There is no
collision check against:
- `ir.state` entries (`<data>` items)
- `ir.computed` entries (`<computed>` items)
- `ir.refs` entries
- user-authored top-level `let`/`const` in the residual script body
- `ir.emits` (which produce `on${e.toLowerCase()}` destructure entries â€” a
  `<emit>onsearch</emit>` plus a slot named `onsearch` would also clash)

Concretely, any component that declares a default `<slot>` AND has any state
named `children` (e.g., `<data> children = [...] </data>` â€” plausible in a
Tree or List component) emits two top-level `const`/`let children` bindings.
The Svelte/TS compiler rejects this with `SyntaxError: Identifier 'children'
has already been declared` and the entire component fails to compile.

The merge also clobbers callers who deliberately declared `let snippets =
...` in script (the `snippets` rest-prop is also unconditionally destructured
whenever `ir.slots.length > 0`).

**Fix:** Either (a) validate at IR-build time that slot names do not
collide with state/computed/ref/emit names (push a new diagnostic, e.g.
`ROZ960 SLOT_NAME_COLLIDES_WITH_BINDING`); OR (b) use private synthetic
names for the `$derived` and let the consumer-facing reference indirect
through it. Example for (b):

```typescript
// emitSlotDerivedMerges
for (const s of ir.slots) {
  const key = s.name === '' ? 'children' : s.name;
  const internal = `__${key}Slot`;  // collision-proof private name
  lines.push(`const ${internal} = $derived(__${key}Prop ?? snippets?.${key});`);
}
// Then update emitSlotInvocation.ts to render {@render __<key>Slot?.(...)}
// instead of {@render <key>?.(...)}
```

The detection-then-error path (a) is the more user-honest choice and
mirrors the existing ROZ200/ROZ947 conventions.

### CR-02: Lit default-slot pre-seed walks `this.children` (Element-only), making text-node tolerance dead code

**File:** `packages/targets/lit/src/emit/emitSlotDecl.ts:117-120`

**Issue:** The emitted pre-seed line for the default slot is:

```js
this._hasSlotDefault = Array.from(this.children).some(
  (el) => !el.hasAttribute('slot') &&
          (el.nodeType !== 3 || (el.textContent?.trim().length ?? 0) > 0)
);
```

`Element.children` returns an `HTMLCollection<Element>` â€” text nodes are
NOT included. Every element in the iterated collection has `nodeType === 1`
(Element node), so the `el.nodeType !== 3` branch is always true and the
text-content fallback is unreachable. The docstring at line 115-117
explicitly promises text-node tolerance ("a text node containing
non-whitespace content"), but the implementation cannot deliver it.

Downstream consequence: a consumer who writes
`<my-component>Just some text</my-component>` (a perfectly valid default-slot
fill â€” the HTML spec projects text nodes into the unnamed `<slot>`)
gets `_hasSlotDefault === false` on first paint. Combined with the typical
conditional wrapper pattern
`${this._hasSlotDefault ? html\`<div><slot></slot></div>\` : nothing}`,
the wrapper element never renders, the inner `<slot>` never gets a host,
and `@queryAssignedElements` (also Element-only) never fires â€” the
chicken-and-egg deadlock that D-LIT-15 was supposed to fix re-emerges for
the text-only default-slot case.

The slotchange handler also reads `_slotDefaultElements` (queryAssigned
**Elements**), so the runtime updater carries the same blind spot â€” text
content never flips `_hasSlotDefault` true.

**Fix:** For the default-slot branch specifically, walk `this.childNodes`
instead of `this.children`:

```js
this._hasSlotDefault = Array.from(this.childNodes).some((node) => {
  // Element: must not carry slot= (otherwise it routes to a named slot)
  if (node.nodeType === 1) {
    return !(node as Element).hasAttribute('slot');
  }
  // TextNode: counts when it has non-whitespace content
  if (node.nodeType === 3) {
    return ((node.textContent ?? '').trim().length) > 0;
  }
  return false;
});
```

Then mirror the same treatment in the slotchange wiring's `update` closure â€”
read `slotEl.assignedNodes({ flatten: true })` (Node-typed, includes text)
instead of `this._slotDefaultElements`, since the latter is Element-only
by design (D-LIT-14 correction).

Named slots are correctly Element-only because `slot=` is an HTML attribute
that only Elements can carry, so the existing emit for named slots is
fine â€” only the default-slot branch needs the childNodes treatment.

### CR-03: Slot names and prop names interpolated into emitted code with zero escaping or validation

**File:** `packages/targets/lit/src/emit/emitSlotFiller.ts:353, 397-398,
452-454, 469-477, 488, 569, 661`
**File:** `packages/targets/lit/src/emit/emitSlotDecl.ts:83, 89, 90, 95, 101,
104, 119, 123, 129`
**File:** `packages/targets/lit/src/emit/emitTemplate.ts:400, 948, 950`
**File:** `packages/targets/svelte/src/emit/emitScript.ts:230-231, 269-271`

**Issue:** Across the Lit and Svelte emitters, author-supplied slot names
(`filler.name`, `slot.name`) and prop names (`attr.name`) are interpolated
into emitted code/attribute strings without any validation or escaping at
any layer. The IR layer (`lowerSlots.ts:108-115, 146-153`) accepts whatever
the parser put in the `name` attribute value verbatim â€” no character-class
check, no identifier-validity check, no schema.

A slot named `foo"]bar` produces:
- `emitSlotFiller.ts:453` â†’ `this.shadowRoot?.querySelector('[slot="foo"]bar"]')`
  â€” CSS selector parse error at runtime (querySelector throws).
- `emitSlotFiller.ts:569` â†’ `<div slot="foo"]bar">â€¦</div>` â€” broken HTML.
- `emitSlotDecl.ts:83` â†’ `private _hasSlotFoo"]bar = false;` â€” invalid TS
  identifier (compile error in the emitted output).

A slot named `2items`:
- `emitSlotDecl.ts:83` â†’ `private _hasSlot2items` â€” fine, but the
  Svelte `emitScript.ts:230` `${key}: __${key}Prop` produces
  `2items: __2itemsProp` â€” invalid destructure-target identifier.

A slot named `foo-bar` (kebab-case, plausible for HTML5 native slot names):
- `emitSlotDecl.ts:83-90` â†’ `_hasSlotFoo-bar` (after PascalCase) â€” invalid TS.
- `emitScript.ts:230, 269-270` â†’ `let { foo-bar: __foo-barProp }` and
  `const foo-bar = $derived(__foo-barProp ?? snippets?.foo-bar)` â€” invalid
  destructure, invalid const name.

Two-way binding prop names face the same gap on `attr.name`:
- `emitTemplate.ts:246-247` â†’ `.${attr.name}=...` and `@${kebab}-change=...`
  â€” a prop name `foo"` produces broken attribute syntax.

The dispatchEvent translation in `emitTemplate.ts:400` is the ONE site that
is safe â€” the regex character class `[A-Za-z0-9_]` on the captured slot/param
groups foreclosures injection â€” but it is an exception, not a contract.
Every other emit site assumes the slot/prop name is a valid JS identifier
without checking.

**Fix:** Add a `validateIdentifierName(name, locationDesc, loc)` helper in
`packages/core/src/semantic/` and call it during `lowerSlots`, lowerProps,
and the lowering of `r-model:propName=` so the IR-time validator (ROZ-series
diagnostic) rejects names that are not valid JS identifiers
(`/^[A-Za-z_$][A-Za-z_$0-9]*$/`) at the source. This pushes the failure to
the user-facing diagnostic surface instead of letting the emitter silently
produce broken cross-target output.

Suggested codes:
- `ROZ961 SLOT_NAME_NOT_IDENTIFIER` â€” slot's `name=` attribute must be a
  valid JS identifier (also doubles as the field-suffix root).
- `ROZ962 PROP_NAME_NOT_IDENTIFIER` â€” already implicit in many places; add
  the explicit check at the `<props>` parser.

The narrow `[A-Za-z_$][A-Za-z_$0-9]*` constraint is appropriate for v1: it
matches HTML's `slot=` valid values, JS destructure patterns, and Lit/Svelte
field identifiers in one rule.

### WR-01: `lvalue.ts` accepts `OptionalMemberExpression`, but downstream resolvers do not â€” silently degrades to defensive emit

**File:** `packages/core/src/semantic/lvalue.ts:62, 109`
**File:** `packages/targets/lit/src/emit/resolveLitSetterText.ts:97-129`
**File:** `packages/targets/react/src/emit/resolveTwoWayTarget.ts:103-138`

**Issue:** `lvalue.ts` deliberately accepts both `MemberExpression` AND
`OptionalMemberExpression` (line 109; docstring example line 62 shows
`$data?.x â†’ { â€¦ depth: 1 }` as a passing case). But the per-target
resolvers downstream are inconsistent:

- `resolveLitSetterText.ts:97-129` â€” Case 1 (`$data.X`) and Case 2
  (`$props.X`) gate on `t.isMemberExpression(expr) && ...` only; an
  optional-chain `$data?.X` falls through to the deep-chain branch
  (Case 3, line 141), which only matches when the chain has TWO levels
  of MemberExpression â€” a one-level OMS doesn't match. It ultimately
  hits the `generate(expr, GEN_OPTS).code` fallback at line 205,
  emitting `$data?.X = e.detail` â€” an assignment to an optional chain,
  which is a SyntaxError in JS.
- `resolveTwoWayTarget.ts (React):103-138` â€” same shape; the case 1
  guard `t.isMemberExpression(expr) && !expr.computed && ...` is
  MemberExpression-only. The defensive throw at line 161 fires
  (`unexpected non-writable LHS reached the React emitter â€” validator
  should have caught this with ROZ951`) â€” except the validator did
  accept it, so the throw is a real runtime crash of the compiler,
  not a defensive guard.
- `resolveTwoWayTarget.ts (Solid):100-129` â€” DOES check both
  `MemberExpression` and `OptionalMemberExpression` (line 100). Asymmetric
  with React/Lit.

In practice users rarely write `$data?.x` as an r-model RHS, so the
real-world incidence is low â€” but the asymmetry is a latent foot-gun.

**Fix:** Either (a) tighten `lvalue.ts` to reject OMS for now (deferred to
v2 with an explicit comment), aligning with the conservative React/Lit
side; OR (b) loosen the React and Lit resolvers to accept OMS and emit
the same setter text as MemberExpression (the runtime semantics are
identical â€” `?.` doesn't appear on the LHS of an assignment, but the
resolver only cares about the property segment names). Option (a) is the
faster, smaller-surface fix; the docstring example at lvalue.ts:62
should be revised in either case.

### WR-03: Lit `tryWire` microtask retry leaks observer when both attempts succeed in rapid succession

**File:** `packages/targets/lit/src/emit/emitSlotFiller.ts:448-464`

**Issue:** The Phase 07.3.1 D-03 microtask retry pattern is:

```js
const tryWire = () => {
  const slotEl = â€¦querySelector(â€¦);
  if (slotEl) {
    const unsubscribe = observeRozieSlotCtx(slotEl, (c) => { â€¦ });
    this._disconnectCleanups.push(unsubscribe);
    this.<wiredFieldName> = true;
  }
};
tryWire();
queueMicrotask(() => { if (!this.<wiredFieldName>) tryWire(); });
```

The `wiredFieldName` flag is set INSIDE `tryWire()` only after `unsubscribe`
is pushed. If both calls execute and both find a `slotEl` (e.g., the
producer-upgrade race resolves between the sync call and the microtask),
both calls register `observeRozieSlotCtx` callbacks AND push their
`unsubscribe` to `_disconnectCleanups`. The microtask's `if (!this.<wired>)`
guard correctly prevents this in the canonical race-resolution path â€”
because the first call sets the flag synchronously. But: the same retry
also runs from `updated()` (lines 470-481) and there is NO mutex between
the two paths. If `firstUpdated()` runs the microtask in cycle N, AND
`updated()` cycle N+1 fires before the microtask resolves (Lit batches
updates across microtasks), both could read `wiredFieldName === false`
and both pass.

In practice the canonical Lit lifecycle orders these such that the race
window is narrow, but the contract is fragile â€” there is no acquire/release
discipline on `wiredFieldName`. A leaked observer ties up a
`_disconnectCleanups` slot, fires the ctx callback twice per slot event,
and (if the consumer's ctx callback triggers `requestUpdate()`) loops.

**Fix:** Set the flag BEFORE calling `observeRozieSlotCtx`, then clear it
on the unsubscribe path if the call throws:

```js
const tryWire = () => {
  if (this.<wiredFieldName>) return;        // re-entry guard
  const slotEl = â€¦querySelector(â€¦);
  if (!slotEl) return;
  this.<wiredFieldName> = true;             // claim before subscribe
  const unsubscribe = observeRozieSlotCtx(slotEl, (c) => { â€¦ });
  this._disconnectCleanups.push(unsubscribe);
};
```

Same shape applies to the `updated()` re-attempt â€” currently the read+write
pair has no atomicity barrier.

### WR-04: `resolveLitSetterText.ts` non-model `$props.X` fallback silently emits invalid assignment

**File:** `packages/targets/lit/src/emit/resolveLitSetterText.ts:124-129`

**Issue:** When a `$props.X` LHS reaches the resolver with `X` not in
`modelProps`, the code returns `this.${xName}` as a "defensive form
anyway" (line 128). The caller then composes `${setterText} = e.detail`,
producing `this.foo = e.detail` for a non-model `foo`. On Lit this writes
to the @property setter that has no `controllable` write path â€” the value
gets clobbered but does not propagate to the upstream consumer (no
CustomEvent dispatch), making the two-way binding silently one-way.

The comment "should have been caught by ROZ951" is correct, but the
defensive emit is misleading â€” the compiled component looks functional
but loses the upstream notification. A user debugging this would have
to discover the missing dispatch by reading source.

Same issue at line 202-205: the generic fallback emits whatever
`generate(expr, GEN_OPTS).code` produces, which for an arbitrary
non-validated expression could be `this.foo()` or `someFn() = â€¦` â€” both
runtime errors at execution time.

**Fix:** Either throw with a clear diagnostic message (matching the React
resolver's behavior at `resolveTwoWayTarget.ts:161`) OR emit a
`/* ROZIE INTERNAL ERROR: validator regression on r-model lvalue â€” please
report */` comment beside the broken assignment so a downstream user sees
the source of truth in the emitted output.

The throw is the more defensive choice â€” a crashed compile is
unambiguously bad and gets reported; a silently malfunctioning component
is a debugging tarpit.

### WR-06: `wrapWithSlotAttribute` self-close detection assumes `[openIdx, closeIdx]` form, breaks on attributes spanning multiple lines

**File:** `packages/targets/lit/src/emit/emitSlotFiller.ts:542-554`

**Issue:** The self-close detection `const isSelfClose = trimmed[openClose - 1] === '/';` correctly identifies `<img />` but the
post-`/>` check `openClose === trimmed.length - 1` requires the trimmed body
to end EXACTLY at the closing `>`. For a self-closing single-root with
trailing whitespace surviving the `trim()` (unlikely but possible if the
emitter inserts a trailing space inside the tag), the check fails and the
body falls into the non-self-closing path, which then tries to find a
matching `</tag>` that doesn't exist â€” leading the spread path to evaluate
the body as multi-root.

More concretely: the regex `/^<([a-zA-Z][\w-]*)\b/` requires the tag to
start at position 0 of `trimmed`. For a single-root body that has any
leading content (a top-level comment, an interpolation `${cond ? html`<x/>`
: nothing}`), the regex fails entirely and the spread path attempts an
ambiguous case. In most realistic consumer-side fill bodies this is fine,
but the heuristic doesn't degrade gracefully for the corner cases.

**Fix:** Add explicit unit-test coverage for: (a) single self-closing
element with one attribute that contains a `>` in a quote
(`<input title="foo>bar" />`); (b) single root preceded by a comment
(`<!-- â€¦ --><div>â€¦</div>`); (c) body starting with an interpolation
(`${cond ? html`<x/>` : nothing}`). The scanner already handles `${...}`
correctly per the D-03 fix at lines 706-713; verify the integration with
`hasSiblingAtTopLevel` and `spreadSlotAttrAcrossTopLevelElements` covers
these.
