# Phase 07.3.1 Plan 07: Lit consumer emits dispatchEvent for function-typed slot-arg handlers (D-LIT-17) Summary

Lit consumer-side `@click="close"` in a scoped slot fill (where `close` is a function-typed slot param) now dispatches a `CustomEvent('rozie-<slot>-<param>')` that the producer's existing host-listener wiring receives end-to-end — closing the JSON-transport gap where `JSON.stringify` in `data-rozie-params` silently drops function values, leaving the consumer's late-binding read of `this._<X>Ctx?.<param>` permanently undefined.

## What Changed

### Task 1 — `emitTemplate.ts buildEventParts` translates exact-shape to dispatchEvent

The Phase 07.3.1 Blocker #3 (D-03) late-binding wrap stayed in place because it's still needed for composite expressions referencing data-typed slot params (e.g. `:disabled="!remaining"` where `remaining` is a data-typed scoped param). A new, narrower detection runs BEFORE the existing wrap:

```js
const dispatchMatch = handlerRaw.match(
  /^\s*this\._([A-Za-z0-9_]+)Ctx\?\.([A-Za-z_][A-Za-z0-9_]*)\s*$/,
);
let handler: string;
if (dispatchMatch) {
  const slot = dispatchMatch[1]!;
  const param = dispatchMatch[2]!;
  handler = `(e) => this.dispatchEvent(new CustomEvent('rozie-${slot}-${param}', { detail: e, bubbles: true, composed: true }))`;
} else {
  // ...existing late-binding wrap...
}
```

The regex is anchored full-string (`^...$`) so any composite or chained expression falls through to the existing wrap. The two capture groups recover the slot name (`<X>` from `_<X>Ctx`) and the param name (`<param>` from `?.<param>`); both are interpolated into a kebab-style event name that matches the producer's `addEventListener('rozie-<slot>-<param>', ...)` wiring (already emitted via `emitHostListenerWiring.ts`).

Inline provenance comment block above the detection explains the architecture: function values can't transit `data-rozie-params` (JSON drops functions), so the producer side already runs an event-based path via host listener; the consumer must dispatch that event instead of trying to invoke the (always-undefined) function.

## Architectural Notes

### Why dispatchEvent vs. fixing the JSON transport

A "simpler" alternative would be to extend the producer's `data-rozie-params` JSON serialization to include function values (e.g., by storing function names + dispatching via a registry). That path was rejected at planning time (per the plan's `<objective>` and the original D-LIT-12 design) because:

1. The producer ALREADY has the host-listener wiring in place (emitted by emitHostListenerWiring.ts at every slot decl with function-typed params). Reusing it means the consumer-side fix is a single-call-site translation.
2. A function registry would require lifecycle management (unmount → deregister, slot unmount → free fn handle) that the CustomEvent path gets for free via shadow-DOM event propagation.
3. `bubbles: true, composed: true` makes the event cross the shadow boundary so the parent custom element (the producer) receives it — exactly the wiring that's already in place.

### Cascade order matters

The two regexes (D-LIT-17 anchored vs. D-03 loose) overlap intentionally: any string matching D-LIT-17 also matches D-03 (since D-03 is just `/this\._[A-Za-z0-9_]+Ctx\?\./`). The cascade runs D-LIT-17 FIRST and only falls through if it fails. Reordering them would break the fix — the loose D-03 wrap would silently swallow every bare-identifier handler, leaving the JSON-drop gap unfixed.

### Composite expression fallback is a known accept-disposition

Per threat register T-07.3.1-17: composite expressions like `close()` (with parens, making it a CallExpression) or `close, doSomething()` (sequence) fall back to the broken late-binding wrap. The wrap is benign for data-typed param composites (they're undefined at first render → no-op, populated by `updated()` retry → works); it's broken for function-typed param composites (always undefined). Accepted because bare-identifier slot-param handlers cover the vast majority of real usage; future work can extend the detection to specific CallExpression shapes if dogfooding surfaces a real pattern.

## Threat Flags

None — no new network endpoints, no new auth surface, no new file/schema access introduced. The dispatchEvent payload (`detail: e` where `e` is the DOM event) crosses the shadow boundary intentionally; the namespaced event name (`rozie-<slot>-<param>`) makes accidental cross-listener collisions vanishingly unlikely (documented in T-07.3.1-18).
