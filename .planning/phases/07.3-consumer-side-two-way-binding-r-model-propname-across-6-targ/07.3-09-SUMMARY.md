# Phase 07.3 Plan 09: Dogfood + acceptance (TWO-WAY-05/06)

## One-liner

Wave 4 dogfood closer: ModalConsumer.rozie and WrapperModal.rozie use the
shipped `r-model:propName=` directive end-to-end (per-instance state +
forwarding pattern); 12 byte-locked dist-parity fixtures regenerated; 4 of
6 close-spec cells now pass after un-fixme; parity.md replaces the
"pending" note with the shipped directive doc.

## What was built

### Source dogfood (examples/*.rozie + 6 per-consumer copies)

`examples/ModalConsumer.rozie` — D-04 per-instance state split:
`$data.open` exploded into `$data.open1`, `$data.open2`, `$data.open3`
(one writable per Modal instance). Each Modal gets its own
`r-model:open="$data.openN"` directive — the named-fill Modal (#1), the
dynamic-name fill Modal (#2), and the WrapperModal (#3). The `onConfirm`
script in the consumer references `$data.open1` (consistent with the
per-instance split).

`examples/WrapperModal.rozie` — D-03 forwarding pattern: wrapper declares
its own `open: { type: Boolean, default: false, model: true }` prop and
forwards via `r-model:open="$props.open"` on the inner Modal. The wrapper
is now itself two-way addressable — consumers can
`r-model:open="$data.x"` on `<WrapperModal>` and the writeback
propagates through the wrapper's controllable-state instance into the
inner Modal.

Propagated to 6 per-consumer copies for byte-identical local consumption:
react-vite, svelte-vite, angular-analogjs, lit-vanilla-demo,
solid-vite (NEW), and vue-vite (NEW). Solid-vite and vue-vite previously
imported the canonical via `../../../../ModalConsumer.rozie` relative
path; per-consumer copies now also exist so a future per-consumer
divergence is structurally permitted without a tree refactor.

## Deviations from plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Wave-2 hasTwoWayBinding short-circuit was never removed**

- **Found during:** Task 2 (bootstrap regen produced 0-byte
  ModalConsumer.* and WrapperModal.* fixtures with no diagnostics)
- **Issue:** `packages/core/src/compile.ts` carried a Plan 02
  `hasTwoWayBinding(ir)` guard that returned empty `result.code`
  whenever ANY `twoWayBinding` IR attribute was present, on the
  rationale that Wave-3 emitters didn't yet handle the new kind.
  All Wave-3 emitter plans (07.3-03..08) shipped their `kind:
  'twoWayBinding'` branches; the guard comment says "removed when the
  last Wave 3 plan lands" — but no plan did. Every valid two-way
  binding silently compiled to 0 bytes.
- **Fix:** Removed the guard + the `hasTwoWayBinding` helper + the
  `_IRComponentForGuard` import alias. Comment block at the call site
  documents why the guard was removed and where the new safety net
  lives (per-target emitter exhaustiveness on AttributeBinding union).
- **Files modified:** packages/core/src/compile.ts
- **Commit:** a42a5e2

**2. [Rule 1 - Bug] Lit's internal `emitAttribute` was missing the twoWayBinding branch**

- **Found during:** Task 2 (regenerated Lit fixtures missed `.open=${...}
  @open-change=${...}` pair on `<rozie-modal>` and `<rozie-wrapper-modal>`)
- **Issue:** Lit has TWO emit helpers: the standalone exported
  `emitTemplateAttribute` (tested in isolation by
  `__tests__/emit-two-way.test.ts`) AND an internal `emitAttribute` inside
  `emitTemplate.ts` that is what actually drives per-element output. Plan 08
  added the twoWayBinding branch only to the standalone helper; the internal
  one silently dropped the attribute.
- **Fix:** Added the symmetric branch to internal `emitAttribute` (same
  shape — `.<prop>=${val} @<kebab>-change=${(e: CustomEvent) => { setter =
  e.detail; }}`) using the already-shared `resolveLitSetterText` +
  `toKebabCase` helpers so byte-equal CustomEvent name parity with the
  producer is preserved by construction.
- **Files modified:** packages/targets/lit/src/emit/emitTemplate.ts
- **Commit:** a42a5e2 (combined with Bug #1)

**3. [Rule 1 - Bug] React forwarding-pattern emit referenced undefined `onPropChange` identifier**

- **Found during:** Task 3 (close-spec VR run for React showed click
  was a no-op; investigation showed WrapperModal.tsx emitted
  `<Modal open={open} onOpenChange={onOpenChange} ... />` where
  `onOpenChange` is an undefined free identifier)
- **Issue:** `packages/targets/react/src/emit/resolveTwoWayTarget.ts`'s
  `$props.X (model: true)` branch returned `setter: on<Cap>Change`,
  intended to reference the consumer's own `on<Cap>Change` callback prop.
  But the React emit pattern accesses props via `props.<name>` and
  doesn't destructure at function top — `onOpenChange` is never a free
  identifier in scope. The correct setter is the LOCAL
  `set<Cap><prop>` returned by the wrapper's own useControllableState
  call (which in turn invokes `props.onOpenChange` via the
  `onValueChange` callback — canonical Radix/shadcn forwarding).
- **Fix:** Changed setter to `set${capitalize(prop)}` (matches the
  `$data.X` branch's local-setter shape); inline comment documents why
  the bare `on<Cap>Change` is wrong (no destructured-prop identifier
  in React emit). Solid was already correct; Vue/Svelte/Angular/Lit
  use compiler macros and don't share this code path.
- **Files modified:** packages/targets/react/src/emit/resolveTwoWayTarget.ts,
  tests/dist-parity/fixtures/WrapperModal.tsx (re-blessed)
- **Commit:** 47767d5

### Deferred (out of scope)

**4. [Scope Boundary] Svelte + Lit close-spec VR cells fail with pre-existing scoped-slot bugs**

After Plan 09's r-model:open= directive wires the two-way emit path in
all 6 targets (verified via dist-parity 336/336 GREEN), the VR close-spec
passes in vue/react/angular/solid but fails in svelte + lit:

- **Svelte:** Producer `Modal.svelte` calls `{@render header(close)}`
  (positional arg, bare `close` function). Consumer declares
  `{#snippet header({ close })}` (destructures from OBJECT). Result: `close` is
  undefined in the snippet body; clicking the × is a no-op. This is a
  Svelte producer-emit bug from Phase 07.2 (originally masked by the
  `.fixme`).
- **Lit:** Modal-1's consumer-fill button has no usable accessible name at
  first paint because the async `observeRozieSlotCtx` registration in
  `firstUpdated()` hasn't fired yet. The `@click=${this._headerCtx?.close}`
  resolves to `undefined` on initial render. This is a Lit producer-emit
  / runtime issue from Phase 06.4 or 07.2-03 (also previously masked by
  the `.fixme`).

Both are independent of consumer-side two-way binding — Plan 09 surfaces
them but does not introduce them. Logged in
`.planning/phases/07.3-.../deferred-items.md` for a Phase 07.4 follow-up
or two atomic Quick fixes. Plan 09 ships with the close-spec un-fixme'd
(structurally correct) but documents the remaining 2 cells as known-failing.
