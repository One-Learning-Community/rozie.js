# Phase 07.3 Plan 03: Vue consumer-side two-way binding emit Summary

Wire `<Modal r-model:open="$data.x">` into Vue 3.4+ idiom `<Modal v-model:open="x">` by replacing the Wave 2 throw-stub in `emitSingleAttr` with a real discriminated-union branch on `AttributeBinding.kind === 'twoWayBinding'`.

## What was built

**Emit branch (5 LoC + comment block) in `packages/targets/vue/src/emit/emitTemplateAttribute.ts`:**

```ts
if (attr.kind === 'twoWayBinding') {
  // Phase 07.3 Wave 3 Plan 07.3-03 (TWO-WAY-03) — consumer-side two-way binding.
  // Mirror native Vue 3.4+ idiom (D-01)…
  const expr = rewriteTemplateExpression(attr.expression, ir);
  return `v-model:${attr.name}="${expr}"`;
}
```

Placed at the top of `emitSingleAttr` (before the static / binding / interpolated branches) — TypeScript exhaustiveness on the discriminated union enforces handling.

**Why the rewriter is sufficient:** Vue's `<template>` context auto-unwraps top-level Refs, so `$data.open1` → `open1` and `$props.open` (when `isModel: true`) → `open` happen via the existing `rewriteTemplateExpression` pass — the same pass every other Vue template attribute already uses. No bespoke setter wiring is needed (Vue's macro does the rest at the producer side via the `defineModel<T>('propName')` machinery shipped in TWO-WAY-02).

**`attrToArraySegment` stub left as a throw** (intentional): `twoWayBinding` is structurally never valid inside a `class`/`style` array merge — the throw documents the contract rather than acting as deferred work.

## Test coverage

New file `packages/targets/vue/tests/emit-two-way.test.ts` with 4 `it()` cases:

1. **Base case** — `$data.open1` → `v-model:open="open1"` (exact string match)
2. **camelCase propName** — `closeOnEscape` → `v-model:closeOnEscape="flag"` (preservation, no kebab-casing)
3. **Forwarding pattern (D-03 permissive LHS)** — `$props.open` on a wrapper with `model: true` prop → `v-model:open="open"`
4. **Regression guard** — bare `r-model="$data.draft"` on form input still emits `v-model="draft"` (TWO-WAY-02 untouched)

All assertions use `expect(result).toBe(...)` against literal strings (not regexes), per the plan's `acceptance_criteria` requirement.
