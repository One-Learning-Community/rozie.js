# Phase 07.3 Plan 07: Solid consumer-side two-way binding emit (TWO-WAY-03) Summary

**Solid target now emits `prop={local()} on<Cap>Change={setter}` for `r-model:propName=` consumer-side two-way binding, with a new `resolveTwoWayTarget` helper that resolves `$data.X` and `$props.X` (model:true forwarding) to bare Accessor + Setter identifiers — Solid's only structural difference from React is the trailing `()` invocation on the Accessor.**

## What Was Built

### Emit branch (`emitTemplateAttribute.ts`)

The Wave 2 stub-throw in the `twoWayBinding` branch is replaced with the real emit:

```ts
if (attr.kind === 'twoWayBinding') {
  const target = resolveTwoWayTarget(attr.expression, ctx.ir);
  if (target === null) {
    // defensive fallback — one-way binding
    return { jsx: `${jsxNameFallback}={${exprCodeFallback}}`, diagnostics };
  }
  const { local, setter } = target;
  const jsxName = colonPropToSolidName(attr.name);
  const eventProp = `on${capitalize(jsxName)}Change`;
  return {
    jsx: `${jsxName}={${local}()} ${eventProp}={${setter}}`,
    diagnostics,
  };
}
```

Key shape (D-01, RESEARCH §Solid lines 179-184):

```
<Modal r-model:open="$data.open1">
  → <Modal open={open1()} onOpenChange={setOpen1} />
```

The Accessor invocation `local()` is the only structural difference from React's emit shape. The Setter is the bare identifier — never invoked at the call site (the runtime's controllable signal pushes the value into `_props.on${Cap}Change?.(v)` itself).

### New helper (`resolveTwoWayTarget.ts`)

Three accepted lvalue shapes (matches the D-03 permissive rule in `core/src/semantic/lvalue.ts`):

| RHS expression | `local` | `setter` | Solid binding source |
|----------------|---------|----------|----------------------|
| `$data.X` | `X` | `setX` | `const [X, setX] = createSignal(...)` (emitScript.ts:152-158) |
| `$data.X.Y.Z` | `X` | `setX` | Root createSignal pair; deep-write is wrapper-side |
| `$props.X` (X has `model: true`) | `X` | `setX` | `const [X, setX] = createControllableSignal(_props, 'X', ...)` (emitScript.ts:126-149) |

The forwarding `$props.X` case is the WrapperModal pattern: a consumer's own `model: true` prop is hoisted into a controllable signal pair with the **same `{ X, setX }` naming** as the `$data` case. The child component (`<Modal>`) just forwards into that pair — no special `local.X` / `local.onXChange` shape needed. This was verified against `emitRModel.ts:60-62` which uses the exact same target-resolution logic for the form-input bare `r-model=` directive.

Returns `null` only for shapes not covered by D-03 (literals, computed refs, `$refs`, etc.) — the IR-time validator rejects these via ROZ951 long before emit runs, so the null branch is a defensive fallback that emits a bare one-way binding rather than crashing.

### Splitprops convention used

This plan's `<behavior>` instruction asked the implementer to verify which Solid splitProps convention applies for the `$props.X` (forwarding) case. **Verification:**

- `emitScript.ts:126-149` shows that `model: true` props are NOT split into `local.*` — they are hoisted into a top-level controllable signal pair via `const [X, setX] = createControllableSignal(_props, 'X', dflt);`.
- `splitProps` in Solid only handles non-model props plus children (see `emitScript.ts` + `shell.ts:165` `splitPropsCall`).
- Therefore the forwarding inner-child binding reuses the controllable pair directly: `open={open()} onOpenChange={setOpen}`. There is no `local.open` Accessor or `local.onOpenChange` callback to thread through.

This is confirmed by `emitRModel.ts:60-62` which uses identical resolution logic for the form-input form.

### Test (`emit-two-way.test.ts`)

Three behavior cases, all asserting exact emit-string contracts:

1. **`$data.open1`** → contains `open={open1()} onOpenChange={setOpen1}`
2. **camelCase propName** (`r-model:closeOnEscape="$data.flag"`) → contains `closeOnEscape={flag()} onCloseOnEscapeChange={setFlag}`
3. **Forwarding `$props.open` (model:true)** → contains `open={open()} onOpenChange={setOpen}`
