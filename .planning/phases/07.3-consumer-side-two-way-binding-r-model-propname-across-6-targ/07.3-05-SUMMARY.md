# Plan 07.3-05 — Angular target consumer-side two-way binding emit

## What was built

The Wave-2 throw-stub in `packages/targets/angular/src/emit/emitTemplateAttribute.ts`'s `emitSingleAttr` is replaced with a real consumer-side `r-model:propName=` emit branch. The branch produces the Angular LONG-FORM signal pair:

```
[propName]="signalName()" (propNameChange)="signalName.set($event)"
```

NOT the banana-in-a-box `[(propName)]` sugar — per RESEARCH §Landmines, some Angular 19.x point releases mis-recognise WritableSignal LHS under the banana form. Mirrors the existing form-input `r-model` branch's `[ngModel]`/`(ngModelChange)` precedent in the same file.

### Emit branch shape

```ts
if (attr.kind === 'twoWayBinding') {
  const signalName = resolveSignalNameForLValue(attr.expression, ctx.ir);
  const bindingName = isComponentTag(ctx) ? kebabToCamel(attr.name) : attr.name;
  if (signalName !== null) {
    return `[${bindingName}]="${signalName}()" (${bindingName}Change)="${signalName}.set($event)"`;
  }
  // Non-signal fallback — rare-case degrade to one-way binding.
  const expr = rewriteTemplateExpression(attr.expression, ctx.ir, {...});
  return `[${bindingName}]="${expr}"`;
}
```

### Signal-resolution helper: extracted (not copied)

The plan's task body offered two options for the signal-name detection logic: **extract** the inline form-input `r-model` block (lines 134-178 of the original file) into a shared helper, OR **copy** it into the new branch with the form-input block untouched if extraction was deemed risky.

**Decision: EXTRACT.** Created `resolveSignalNameForLValue(expr, ir): string | null` as a file-private helper. It encapsulates the `$data.X` / `$props.X` (model:true) MemberExpression-pattern detection. Both the form-input `r-model` branch AND the new Phase 07.3 twoWayBinding branch now call the helper.

**Why extraction was safe:**

- The helper is a pure shape detector — same inputs produce same `signalName | null` output as the inlined version.
- The 60 prior Angular tests (including the load-bearing `SearchInput` form-input shape-lock at `emitTemplate.test.ts` lines 92-107 — `[ngModel]="query()"` / `(ngModelChange)="query.set($event)"`) all still pass.
- Net code shrinks: the original 30-line inline detection block in the form-input branch collapses to a single `resolveSignalNameForLValue(attr.expression, ctx.ir)` call.

### attrToArraySegment stub preserved (intentional)

The second `twoWayBinding` throw-stub in `attrToArraySegment` (used for class/style array merge) is intentionally kept. `twoWayBinding` is a component-prop binding and never participates in class/style aggregation — reaching that branch would be an upstream IR bug, so the throw is the correct guard.
