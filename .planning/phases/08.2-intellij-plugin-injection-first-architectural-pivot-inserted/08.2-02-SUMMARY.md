## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking issue] `RozieMultiHostInjector` emitted one HTML injection per TEMPLATE_BODY token instead of one coalesced injection per template body**

- **Found during:** Task 3 (running `RozieXmlExtensionTest` after Task 2's compile-clean Kotlin source + Task 3's plugin.xml registration — all 6 positive tests failed with `assertNotNull("$attr should have a non-null descriptor", null)` despite the provider being correctly registered).
- **Issue:** Plan 01 SUMMARY claims `IN_TEMPLATE_BODY` collapsed to "a single contiguous TEMPLATE_BODY token" — but the actual JFlex rule pair (`[^<]+ { return TEMPLATE_BODY; }` + a bare `<` rule also returning `TEMPLATE_BODY`) emits the body as a **run** of TEMPLATE_BODY tokens separated by single-character `<` tokens. Empirical lexer dump for `r-if-recognised.rozie`:

  ```
  TOK 9  [39-42] TEMPLATE_BODY '\n  '
  TOK 10 [42-43] TEMPLATE_BODY '<'
  TOK 11 [43-75] TEMPLATE_BODY 'div r-if="$data.visible">visible'
  TOK 12 [75-76] TEMPLATE_BODY '<'
  TOK 13 [76-82] TEMPLATE_BODY '/div>\n'
  ```

  The Plan-01 `RozieMultiHostInjector` had a simple `for (i in tokens.indices)` loop calling `injectHtml(host, tok.range)` for **each** TEMPLATE_BODY token, producing 5 isolated HTML injections per template body. Each injection saw a fragment like `div r-if="$data.visible">visible` (without the `<` prefix or `</div>` closing) — un-parseable as HTML, so the HTML PSI collapsed the interior into `XmlToken:XML_DATA_CHARACTERS` with NO `XmlTagImpl` / `XmlAttributeImpl` produced. The descriptor provider then had no `XmlTag` context to operate on (`PsiTreeUtil.getParentOfType(injected, XmlTag::class.java)` returned null), `getAttributeDescriptor(name, null)` short-circuited to null, every positive test failed.

  This is a latent bug from Plan 01 that didn't manifest until Plan 02 first tried to walk the injected HTML PSI for an `XmlAttribute`. Plan 01's smoke test (`testTemplateBodyIsHtmlInjected`) only asserted "any HTML injection covers offset X" — which passed even with broken per-fragment injections — and gave no signal about whether the injected HTML was structurally parseable.

- **Fix:** Rewrote `RozieMultiHostInjector.getLanguagesToInject` from a per-token `for` loop into a manual `while (i < tokens.size)` loop that, on encountering a TEMPLATE_BODY token, walks forward over the contiguous TEMPLATE_BODY run and emits ONE `injectHtml(host, TextRange(firstStart, lastEnd))` for the whole run. JS / CSS / Style injections retain the per-token shape (their bodies are already single tokens). 28 LOC delta (+22 new, –5 old, +1 comment-block).
- **Files modified:** `tools/intellij-plugin/src/main/kotlin/js/rozie/intellij/injection/RozieMultiHostInjector.kt`
- **Verification:** Empirical PSI dump after the fix shows the expected HTML tree:
  ```
  HtmlTagImpl(HTML_TAG) text='<div r-if="$data.visible">visible</div>'
    XmlAttributeImpl(XML_ATTRIBUTE) text='r-if="$data.visible"'
      XmlTokenImpl(XML_NAME) text='r-if'
  ```
  `RozieXmlExtensionTest` goes 1/7 → 7/7 GREEN. Full test suite (`./gradlew test --rerun-tasks`) exits 0 with 26 tests passing across 5 test classes (RozieXmlExtensionTest 7, RozieLexerTest 7, RozieSyntaxHighlighterTest 3, RozieInjectionTest 8, plus the existing tests).
- **Committed in:** `85485eb` (Task 3 commit, bundled with the descriptor-provider creation + plugin.xml registration because the injector fix is part of what turns the suite GREEN).
- **Plan-modification note:** `RozieMultiHostInjector.kt` was NOT in Plan 02's `files_modified` frontmatter, but the change is necessary for Plan 02's done-criteria to hold. This is the canonical Rule-3 case: pre-existing file outside the plan's enumerated edits whose behavior blocks the plan's goal.
